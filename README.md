# multitoolforge.github.io

The central landing page and project directory for
[Multitool Forge](https://multitoolforge.org) — an Apereo Foundation
open-source community building LTI®-compliant tools to fill gaps in enterprise
Learning Management Systems.

---

## Tech Stack

| Layer                 | Technology                              | Version           |
| --------------------- | --------------------------------------- | ----------------- |
| Runtime               | [Deno](https://deno.com)                | 2.x               |
| Static Site Generator | [Lume](https://lume.land)               | 3.2.2             |
| Templating            | [Vento](https://vento.js.org) (`.vto`)  | bundled with Lume |
| CSS                   | [Tailwind CSS](https://tailwindcss.com) | v4                |
| Deployment            | GitHub Pages via GitHub Actions         | —                 |

### Lume Plugins in Use

| Plugin           | Purpose                                                              |
| ---------------- | -------------------------------------------------------------------- |
| `tailwindcss`    | Processes CSS files registered with `site.add()` through Tailwind v4 |
| `attributes`     | Allows setting HTML attributes from front matter                     |
| `code_highlight` | Syntax highlighting for fenced code blocks                           |
| `nav`            | Builds navigation trees from page structure                          |
| `base_path`      | Rewrites URLs relative to the site location                          |
| `resolve_urls`   | Ensures all URLs resolve correctly                                   |
| `slugify_urls`   | Normalizes page URLs to lowercase-with-dashes                        |

## Repository Layout

```
multitoolforge.github.io/
├── _config.ts                  # Lume site configuration — plugins, file handling
├── deno.json                   # Deno project config — imports, tasks, permissions
├── styles.css                  # Tailwind CSS entry point + custom component styles
│
├── index.vto                   # Main landing page (hero, mission, projects table, footer)
│
├── _data/
│   └── repos.yml               # Project registry — one entry per repo in the directory
│
├── _includes/
│   └── layouts/
│       └── main.vto            # Base HTML layout (wraps all pages)
│
├── assets/
│   └── logo.svg                # Multitool Forge logo mark (also used as favicon)
│
├── _site/                      # Build output — generated by Lume, not committed
│
└── .github/
    └── workflows/
        └── deploy.yml          # CI/CD: builds the site and deploys to GitHub Pages
```

### Key Concepts

**Content files** are any files not prefixed with `_`. Lume transforms them into
pages in `_site/`.

**Front matter** (the `---` block at the top of `.vto` and `.md` files) sets
per-page variables like `layout` and `title` that are available in templates.

**Layouts** live in `_includes/layouts/`. The `main.vto` layout receives
`{{ content }}` (the rendered page body) and any front matter variables such as
`{{ title }}`.

**`_data/repos.yml`** is automatically exposed by Lume as the `repos` variable
in all templates. The projects table in `index.vto` loops over it with
`{{ for repo of repos }}`.

## Local Development

### Prerequisites

- [Deno 2.x](https://deno.com/manual/getting_started/installation)

### Running the dev server

```sh
deno task serve
```

Opens a live-reloading server at `http://localhost:3000`. Changes to any source
file rebuild and reload the browser automatically.

### Building for production

```sh
deno task build
```

Outputs the static site to `_site/`.

## Adding a Project to the Directory

Edit `_data/repos.yml` and append a new entry:

```yaml
- name: my-lti-tool
  description: A short description of what the tool does.
  type: Tool # Hub | Library | Tool | Plugin
  frameworks:
    - Deno
    - Oak
  year: 2026
  license: MIT
  docs: "https://docs.multitoolforge.org/my-lti-tool"
  repo: "https://github.com/MultitoolForge/my-lti-tool"
  status: active
```

The site will automatically include the new row in the projects table on the
next build. No template changes are needed.

### Field Reference

| Field         | Required | Description                                        |
| ------------- | -------- | -------------------------------------------------- |
| `name`        | yes      | Repo/project name (displayed as the link text)     |
| `description` | yes      | One-sentence description shown in the table        |
| `type`        | yes      | One of: `Hub`, `Library`, `Tool`, `Plugin`         |
| `frameworks`  | yes      | List of frameworks/runtimes used                   |
| `year`        | yes      | Year the project was started                       |
| `license`     | yes      | SPDX license identifier (e.g. `MIT`, `Apache-2.0`) |
| `docs`        | yes      | URL to the project's documentation or homepage     |
| `repo`        | yes      | URL to the GitHub repository                       |
| `status`      | no       | `active`, `archived`, or `incubating`              |

## Adding a Framework Filter Button

The framework filter buttons in `index.vto` are hardcoded. When a new framework
appears in `repos.yml` that isn't already represented, add a button to the
`#framework-filters` group:

```html
<button class="filter-btn" data-group="framework" data-filter="Oak">Oak</button>
```

## Deployment (GitHub Actions)

Every push to `main` triggers the `Deploy Multitool Forge Hub` workflow defined
in `.github/workflows/deploy.yml`.

### Workflow steps

```
push to main
    │
    ▼
[build job]
  1. Checkout repository
  2. Install Deno 2.x
  3. Run `deno task build`  →  generates _site/
  4. Upload _site/ as a GitHub Pages artifact
    │
    ▼
[deploy job]  (runs after build succeeds)
  5. Deploy artifact to GitHub Pages
    │
    ▼
  Live at https://multitoolforge.org
```

### Required repository settings

For the workflow to succeed, GitHub Pages must be configured to deploy via
GitHub Actions:

1. Go to **Settings → Pages**
2. Set **Source** to `GitHub Actions`

The workflow requires these permissions (already set in `deploy.yml`):

| Permission        | Reason                            |
| ----------------- | --------------------------------- |
| `contents: read`  | Checkout the repository           |
| `pages: write`    | Publish to GitHub Pages           |
| `id-token: write` | OIDC token for trusted deployment |

## Styles

`styles.css` is the Tailwind entry point. It uses the Tailwind v4 import syntax:

```css
@import "tailwindcss";
```

Custom component styles (e.g. `.filter-btn`) are written as plain CSS selectors
beneath the import — avoid `@apply` with Tailwind v4 as it requires additional
`@reference` directives.
